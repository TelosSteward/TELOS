# TELOS Repository Map and Component Overview

A complete overview of the TELOS runtime governance framework, including directory layout, core components, and validation structure.

⸻

## Core Purpose

TELOS provides runtime governance for LLM sessions using a mathematical attractor derived from declared purpose, scope, and boundaries.
It measures semantic drift, computes proportional corrections, and exports per-turn telemetry for comparative validation.

This document outlines the repository layout, describes the function of each component, and maps how data flows through the system.

## Core Modules

|File                      |Description                                                                                                                                          |
|--------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------|
|primacy_math.py           |Mathematical substrate of TELOS — τ-weighted attractor center, basin radius (r = 2/max(ρ,0.25)), Lyapunov stability, and fidelity metrics.           |
|intervention_controller.py|Decides when to apply proportional interventions based on drift thresholds (εₘᵢₙ, εₘₐₓ); triggers reminders, regenerations, or anti-meta corrections.|
|unified_steward.py        |Orchestrates runtime loop. Integrates conversation manager, embedding provider, attractor math, controller, and health monitor; exports telemetry.   |
|embedding_provider.py     |Provides deterministic and semantic embedding utilities for drift measurement.                                                                       |
|conversation_manager.py   |Maintains conversation history and handles context-window trimming.                                                                                  |

## Validation Infrastructure

|File                    |Description                                                                                                                                                       |
|------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|baseline_runners.py     |Defines five governance modes — Stateless, Prompt-Only, Cadence, Observation, and TELOS Runtime — for comparative validation.                                     |
|system_health_monitor.py|Runtime diagnostics system (“EKG of TELOS”) - detects NaN values, config errors, threshold violations, and anomalies. Exports health reports and diagnostic plots.|
|run_internal_test0.py   |Executes Internal Test 0 across all five modes; exports telemetry (CSV + JSON summaries).                                                                         |
|telemetry_utils.py      |Handles export of per-turn telemetry and session-level summaries using standardized schemas.                                                                      |
|retro_analyzer.py       |Performs retrospective analysis on completed conversations, applying TELOS math post-hoc for counterfactual validation.                                           |
|run_validation.py       |Generalized study harness for Pilot 0 and later comparative experiments.                                                                                          |

## Five Validation Modes

|Mode         |Description                         |Purpose                                 |
|-------------|------------------------------------|----------------------------------------|
|Stateless    |No governance applied.              |Establishes natural drift baseline.     |
|Prompt-Only  |Single declaration at start.        |Tests declarative persistence.          |
|Cadence      |Re-inject governance every N turns. |Tests fixed-interval correction.        |
|Observation  |TELOS math active, no interventions.|Tests drift detection and fidelity math.|
|TELOS Runtime|Full adaptive governance.           |Tests proportional mathematical control.|

## Data Flow Summary

```
User Input
   ↓
Conversation Manager
   ↓
LLM Client → Raw response
   ↓
Primacy Attractor Math → Drift, Error, Lyapunov
   ↓
Intervention Controller → Decision (monitor / remind / regenerate)
   ↓
Health Monitor → System diagnostics (passive)
   ↓
Unified Steward → Telemetry export
   ↓
Validation Runners → Aggregated results
```

⸻

## Generated Artifacts

After each run:

- `*_turns.csv` → Per-turn telemetry (error, fidelity, drift)
- `*_summary.json` → Session metrics (fidelity averages, intervention counts)
- `system_health_*.json` → Health diagnostics (events, vital signs, suggestions)
- `health_plot_*.png` → Visual diagnostics (fidelity, error, Lyapunov, events)
- `comparative_summary.json` → Aggregated study results

⸻

## Developer Utilities

Makefile commands:

- `make run` → Launch governed session
- `make validate` → Run comparative validation
- `make smoke` → Quick test suite
- `make clean` → Remove logs and cache

⸻

## Research Positioning

TELOS moves alignment from declaration to measurement.
It enables purpose persistence to be treated as an empirical, testable property of LLM sessions—where governance fidelity, stability, and drift become quantitatively observable rather than assumed.
