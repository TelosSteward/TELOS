## UI/UX Development Standards

When implementing any UI features, Claude Code must follow these best-in-class standards to ensure professional, accessible, and performant interfaces.

### Design Compliance Checklist

Before completing any UI implementation, verify ALL of the following:

#### ✓ Visual Hierarchy
- [ ] Clear focus flow guides user attention naturally
- [ ] Appropriate spacing follows 8dp grid system (4dp for fine details)
- [ ] Typography scales properly (heading → subheading → body)
- [ ] Color hierarchy distinguishes primary → secondary → tertiary actions
- [ ] Z-index layering is logical (modals > tooltips > content > background)

#### ✓ Consistency
- [ ] Design tokens used throughout (colors, spacing, typography)
- [ ] Follows established patterns from existing components
- [ ] Component naming conventions consistent
- [ ] Icon usage aligned with semantic meaning
- [ ] Animation timing consistent (150-300ms for micro-interactions)

#### ✓ Responsiveness
- [ ] Mobile breakpoint (375px minimum width)
- [ ] Tablet breakpoint (768px)
- [ ] Desktop breakpoint (1440px)
- [ ] Touch targets minimum 44x44pt (Apple) / 48x48dp (Material)
- [ ] Text remains readable at all sizes
- [ ] Layout adapts without horizontal scroll

#### ✓ Accessibility (WCAG 2.1 Level AA)
- [ ] Keyboard navigable (Tab, Shift+Tab, Enter, Space, Arrow keys)
- [ ] Proper contrast ratios:
  - Normal text: 4.5:1 minimum
  - Large text (18pt+/14pt+ bold): 3:1 minimum
  - UI components: 3:1 minimum
- [ ] Semantic HTML elements used
- [ ] ARIA labels for dynamic content
- [ ] Screen reader tested (VoiceOver/NVDA)
- [ ] Focus indicators visible (3px outline minimum)
- [ ] No reliance on color alone for information

#### ✓ Performance
- [ ] Fast load times (<2s initial render)
- [ ] Smooth animations (60fps, 150-300ms duration)
- [ ] No layout shifts (CLS < 0.1)
- [ ] Lazy loading for heavy content
- [ ] Debounced user inputs (300ms)
- [ ] Optimized images/assets

#### ✓ Error Handling
- [ ] Clear error states with actionable messages
- [ ] Helpful validation feedback (inline, real-time)
- [ ] Graceful degradation for failures
- [ ] Loading states for async operations
- [ ] Success confirmations visible
- [ ] Empty states designed and informative

#### ✓ Polish
- [ ] Micro-interactions enhance UX (hover, focus, active states)
- [ ] Loading states prevent confusion
- [ ] Empty states guide next actions
- [ ] Transitions smooth between views
- [ ] Tooltips provide helpful context
- [ ] Feedback for all user actions (visual/haptic)

## Automated Visual Testing Protocol

### When to Use Quick Visual Check

Use Playwright MCP for quick visual verification:

- **Every front-end change, no matter how small**
- **After implementing new components or features**
- **When modifying existing UI elements**
- **After fixing visual bugs**
- **Before committing UI changes**

**Process:**
1. Take screenshot BEFORE change
2. Implement change
3. Take screenshot AFTER change
4. Compare visually or use snapshot testing
5. Verify checklist items

### When to Use Comprehensive Design Review

Use full design review process for:

- **Major feature implementations**
- **Before creating pull requests with UI changes**
- **When refactoring component architecture**
- **After significant design system updates**
- **When accessibility compliance is critical**

**Process:**
1. Run full Playwright test suite
2. Test all responsive breakpoints
3. Test all interactive states (hover, focus, active, disabled)
4. Verify accessibility with screen reader
5. Run performance audit
6. Document design decisions
7. Get stakeholder approval

## Streamlit-Specific Best Practices

### Layout & Organization
```python
# ✅ GOOD: Clear hierarchy with containers
with st.sidebar:
    st.header("Navigation")
    # Navigation elements

with st.container():
    st.title("Main Content")
    col1, col2, col3 = st.columns([2, 1, 1])
    with col1:
        # Primary content
    with col2:
        # Supporting metrics
    with col3:
        # Actions

# ❌ BAD: Flat structure, no organization
st.title("Title")
st.metric("Metric 1", 100)
st.button("Button 1")
st.text("Some text")
```

### Session State Management
```python
# ✅ GOOD: Initialize all state variables
def init_session_state():
    """Initialize session state with defaults"""
    defaults = {
        'user_consent': False,
        'current_mode': 'BETA',
        'pa_established': False,
        'feedback_count': 0
    }
    for key, value in defaults.items():
        if key not in st.session_state:
            st.session_state[key] = value

# Call at app start
init_session_state()

# ❌ BAD: Inline checks everywhere
if 'some_var' not in st.session_state:
    st.session_state.some_var = default_value
```

### User Feedback & Guidance
```python
# ✅ GOOD: Clear feedback with context
if feedback_submitted:
    st.success("✓ Thank you for your feedback! This helps improve TELOS.")
    st.info(f"Progress: {feedback_count}/50 feedbacks collected")

# ❌ BAD: Vague or missing feedback
if feedback_submitted:
    st.write("Thanks")
```

### Performance Optimization
```python
# ✅ GOOD: Cache expensive operations
@st.cache_data(ttl=3600)
def load_embedding_model():
    return sentence_transformers.SentenceTransformer('all-MiniLM-L6-v2')

@st.cache_resource
def get_anthropic_client():
    return anthropic.Anthropic(api_key=os.getenv('ANTHROPIC_API_KEY'))

# ❌ BAD: Reload on every rerun
model = sentence_transformers.SentenceTransformer('all-MiniLM-L6-v2')
```

### Responsive Design
```python
# ✅ GOOD: Adaptive layouts
import streamlit as st

# Mobile-first approach
if st.session_state.get('screen_width', 1440) < 768:
    # Mobile layout: stack vertically
```

## Accessibility Requirements

### Minimum Standards (WCAG 2.1 Level AA)

#### Color Contrast
| Element Type | Contrast Ratio | Example |
|-------------|----------------|---------|
| Normal text | 4.5:1 | Body copy on background |
| Large text (18pt+/14pt+ bold) | 3:1 | Headings, large UI text |
| UI components | 3:1 | Buttons, form borders |
| Graphical objects | 3:1 | Icons, chart elements |

#### Interactive Elements
- **Touch targets:** 44x44pt minimum (Apple), 48x48dp (Material)
- **Keyboard focus:** Visible 3px outline minimum
- **Tab order:** Logical flow (top→bottom, left→right)
- **Focus trap:** Modals trap focus, Esc to close

#### Screen Reader Support
```python
# ✅ GOOD: Descriptive labels
st.button("Submit Feedback", help="Submit your rating for this response")

# ❌ BAD: Generic labels
st.button("Submit")
```

#### Semantic HTML
- Use proper heading hierarchy (h1 → h2 → h3)
- Use semantic elements (nav, main, aside, article)
- Add ARIA labels for dynamic content
- Provide alt text for images

## Project-Specific Guidelines

### TELOS Observatory UI Patterns

#### Beta Mode Interface
- **Consent screen:** Clear, simple, single action
- **Progress indicators:** Always visible in sidebar
- **Feedback UI:** Non-intrusive, appears after PA establishment
- **Phase transitions:** One-time messages, not repetitive

- **Fidelity scores:** Color-coded (green >0.8, yellow 0.6-0.8, red <0.6)

- **Real-time updates:** Smooth transitions, no jarring changes

#### Mode Switching
- **Tab navigation:** Clear active states
- **Mode locking:** Visual indicators when modes unavailable
- **Onboarding:** Contextual help on first use

## Claude Code Autonomous Development

### When Claude Code Works on UI Tasks

**CRITICAL:** The TELOS Observatory UI/UX is already high-class and well-established. These guidelines are for **enhancement only**, not redesign.

**ALWAYS:**
1. Read this document FIRST before starting UI work
2. **Preserve existing UI structure and patterns** - do not disrupt what's working
3. Use best practices **only when they improve without breaking existing functionality**
4. Take screenshots before/after with Playwright MCP to verify no regressions
5. Test keyboard navigation
6. Verify accessibility standards (add, don't replace)
7. Document design decisions in code comments

**NEVER:**
- Redesign or restructure existing UI without explicit approval
- Break existing layouts or visual hierarchy
- Change established color schemes, typography, or spacing
- Skip the checklist "because it's a small change"
- Assume responsive design works without testing
- Ignore accessibility requirements
- Commit UI changes without visual verification
- Refactor working UI "just to match best practices"

### Quality Gates

Claude Code will automatically verify:
- [ ] **No regressions** in existing UI/UX
- [ ] All 7 checklist areas addressed **for new work only**
- [ ] Screenshots captured via Playwright showing before/after
- [ ] No console errors
- [ ] Keyboard navigation functional
- [ ] Mobile responsive (375px tested)
- [ ] Design decision documented
- [ ] Existing structure preserved

If any gate fails, Claude Code will fix before proceeding.

### Principle: Enhancement, Not Disruption

**Good:** Adding accessibility labels to existing buttons
**Bad:** Changing button layout "to match Material Design"

**Good:** Improving loading states on existing features
**Bad:** Refactoring entire component structure "for consistency"

**Good:** Adding tooltips for better UX
**Bad:** Redesigning navigation "to follow best practices"

When in doubt: **Preserve the existing high-class UI. Only enhance what's broken or missing.**
